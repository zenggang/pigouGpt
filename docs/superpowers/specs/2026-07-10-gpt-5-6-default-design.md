# GPT-5.6 默认模型设计

## 目标

Pigou AI Console 新增 GPT-5.6，并将其设为所有用户升级后的默认模型。界面展示名称为 `GPT-5.6`，实际向 SUB2API 发送的模型标识为 `gpt-5.6-sol`。

## 已确认约束

- 当前 SUB2API 的 `/models` 接口明确提供 `gpt-5.6-sol`、`gpt-5.6-terra` 和 `gpt-5.6-luna`，但未列出裸别名 `gpt-5.6`。
- 使用能力最强且已明确可用的 `gpt-5.6-sol`，不依赖代理对裸别名的兼容行为。
- 保留 `gpt-5.5` 和 `gpt-5.4` 供用户手动切换。
- 所有已有用户在升级后首次打开网站时统一切换到 GPT-5.6；之后用户再次手动选择其他模型时，选择仍可持久化。
- 不调整提示词、推理强度选项和请求 API 结构。

## 实现设计

### 模型定义与请求校验

在前端模型类型、共享服务端类型、聊天请求白名单以及 ECS 图片任务校验中加入 `gpt-5.6-sol`。模型排序为 GPT-5.6、GPT-5.5、GPT-5.4。服务端非法模型提示同步包含三个受支持模型。

选择器为 `gpt-5.6-sol` 提供用户可读标签 `GPT-5.6`，但请求体和数据库继续保存精确上游模型标识，避免展示名称进入业务数据。

### 默认值与数据库兼容

前端初始模型、设置读取回退值、服务端新会话返回值和数据库建表默认值改为 `gpt-5.6-sol`。

创建会话时显式写入模型字段，不再依赖数据库列默认值。这样即使线上已有表的默认约束仍是 `gpt-5.5`，新会话也会立即、可靠地保存 `gpt-5.6-sol`，无需为本次功能执行高风险的在线表结构变更。历史会话数据不批量改写；用户下一次发送消息时，现有逻辑会把该会话使用的模型更新为当前选择。

### 现有用户一次性迁移

启用新版设置存储键。首次读取新版设置时，如果只有旧版设置，则保留其中合法的推理强度，但忽略旧模型并使用 `gpt-5.6-sol`。新版设置写入后，用户后续手动切换到 GPT-5.5 或 GPT-5.4 仍会正常保存，不会每次刷新都被重置。

### 文档

README 的当前可用模型和默认模型说明更新为 GPT-5.6。原始产品规格中的历史验证记录不批量改写，避免把过去使用 GPT-5.5 完成的验证误写成 GPT-5.6。

## 错误处理与日志

现有聊天请求校验和图片任务参数校验继续拒绝白名单外模型。请求、会话保存和图片任务已有的异常日志链路保持不变；本次不引入新的跨系统调用或降级分支，因此不新增重复日志。

## 测试与验收

先增加专项验证并确认它在实现前因缺少 GPT-5.6 支持而失败，再完成最小实现。验收覆盖：

1. 前端默认值和设置回退值均为 `gpt-5.6-sol`。
2. 旧设置只迁移推理强度，不恢复旧模型。
3. 选择器展示 `GPT-5.6`，并仍保留 GPT-5.5、GPT-5.4。
4. 服务端聊天白名单和 ECS 图片任务校验均接受 `gpt-5.6-sol`。
5. 新会话显式保存并返回 `gpt-5.6-sol`。
6. 新建数据库的 schema 默认值为 `gpt-5.6-sol`。
7. README 与当前行为一致。
8. 专项验证、现有验证脚本、ESLint 和生产构建全部通过。

## 非目标

- 不启用 `gpt-5.6-terra`、`gpt-5.6-luna`、`max` 推理强度或 Pro 模式。
- 不批量更新历史会话记录。
- 不重构现有模型配置架构。
- 不修改 SUB2API、鉴权、计费或部署配置。
